🛠️ The Architectural Takeover & Refactoring Story

The Challenge: I took over a Next.js B2B steel trading data portal that was built on a chaotic architecture. The application stored transactional data in a local SQLite file (`data/dev.db`), had image files uploaded directly to local disk folders, and was deployed manually via SFTP. Deployments required SSHing into the host, causing service disruption. Worse, the team was utilizing administrative Root User access keys for deployment scripting.

Performance & Cost Objectives: Although the client's application operates on a small scale, they required the infrastructure to reliably handle peak traffic spikes of 100+ concurrent active users during trade announcements. At the same time, I needed to keep cloud operations highly cost-effective, limiting expenditures to around $100/month. Because the portal is heavily read-oriented, focusing on specifications and price search queries, I designed a multi-layer caching system to shield the database and maximize throughput.

The Refactoring Action Plan: My intervention reorganized the project's foundation into an optimized, secure cloud ecosystem:

  • Database Migration & Query Caching: Refactored the database layer from SQLite to AWS RDS PostgreSQL. Since the platform's profile is query-heavy (steel data catalogs and specifications), I implemented an edge caching strategy using Cloudflare CDN alongside Next.js data cache handlers. This offloaded over 90% of repeat query requests, keeping database CPU utilization low and allowing the entire system to run smoothly on a cost-effective db.t4g.medium (or micro) PostgreSQL database without needing expensive multi-AZ replicas.
  • ECS Container Orchestration: Built a multi-stage Docker configuration targeting production optimizations. Containerized the Next.js runtime, reducing image footprint by 65%. I provisioned an AWS ECS cluster running on Fargate (serverless containers) sitting safely in a private VPC subnet group. Tasks were sized strictly to 0.25 vCPU and 0.5 GB RAM, keeping running container costs to a fraction of traditional server hosting.
  • Zero-Downtime Infrastructure: Configured an Application Load Balancer (ALB) to handle traffic routing and execute rolling deployment upgrades. When GitHub Actions pushes a new version, ECS boots new containers in parallel, performs target group health check polling, shifts user traffic smoothly, and terminates the old revisions without a single dropped packet.

Post-Handoff Security Hardening: To secure the infrastructure, I moved the client away from dangerous long-lived access keys and the AWS root account. I configured fine-grained AWS IAM Policies and created a secure OpenID Connect (OIDC) identity provider. This allowed GitHub Actions to assume a temporary, short-lived IAM role on demand using cryptographically verified web identity tokens. After handoff, all administrator root access was safely locked down, protecting client assets and validating professional DevOps compliance.
